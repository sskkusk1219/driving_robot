# driving_robot - アイデアメモ

## このドキュメントの位置づけ

このドキュメントは壁打ち・ブレストの成果物で、正式な仕様書ではありません。

## プロジェクトの目的
- シャシダイナモの上で車速トレースをするロボットを作製する

## 作りたいもの
- 運転モデルによって導いた基準車速に対するアクセル開度とブレーキ開度をフィードフォワード制御によってアクチュエータを動かし、
  基準車速と実車速のずれをフィードバック制御で調整する運転ロボット

## 解決したい課題
1. **人の運転からの脱却**
   - WLTP, NEDC, UDDS, AMAなどの運転サイクルは人が運転しているため、同じ条件で走行しても実車速にばらつきが生じる。
   - 長時間の人の運転は疲労や集中力の低下によってばらつきが大きくなる。

## ターゲットユーザー

### ペルソナ1: 実験エンジニア

- **年齢**: 25-50歳
- **職業**: 自動車メーカーや研究機関の実験エンジニア
- **経験**: シャシダイナモでの走行試験経験があるが、運転ロボットの操作は初めて
- **今困ってること**:
  - 毎日、同じ運転サイクルを人が運転するのは疲れるし、ばらつきも大きい
  - 運転ロボットを導入したいが、操作方法が複雑そうで不安
  - 運転をロボットに任せて、実験の準備やデータ分析に集中したい
  - もっと正確に基準車速をトレースしてほしい

- **欲しいもの**:
  - 基準車速に対して、正確にアクセルとブレーキを制御してくれるロボット
  - ユーザーフレンドりなGUI
  - 基本機能はシンプル
  - ログ機能
  - タイムスケジュールでペダルの動きを自動化できる機能
  - アクチュエータをGUIから操作して、ペダルを動作させる機能。手動操作みたいなもの
  - 動作が速い

### ペルソナ2: 実験手法開発エンジニア

- **年齢**: 25-50歳
- **職業**: 実験手法を開発するエンジニア
- **経験**: 効率的な実験手法を日々開発しているエンジニア。機械設計、制御、プログラミングの有識者
- **今困ってること**:
　-  運転ロボットのカスタマイズ性が低いと、実験手法の開発に制約が生じる
  -　運転ロボット、シャシダイナモ、計測器を連携させるのが大変

- **欲しいもの**:
  - 運転ロボット、シャシダイナモ、計測器を簡単に連携させるためのAPIやドキュメント
  - GUIでロボットの動作を細かく制御したい
  - ロボットの動作を細かく制御できるインターフェース
  - ロボットの動作をリアルタイムでモニタリングできる機能
  - ロボットの動作をカスタマイズできる柔軟性

## KPI
- 必要に応じて追加したい(ひとまずは下記の要件を満たしたい)
- **Must**
    - 基準車速に対して、実車速が±0.2km/h以内でトレースできること
    - 基準車速に対して、オーバー/アンダーシュートを繰り返さない運転をすること
    - 10時間以上の連続走行ができること
    - 走行までなら説明書なしでもGUIで操作できること
    - 複数の運転サイクル（WLTP, NEDC, UDDS, AMAなど）を再生できること
    - 各車両の設定を保存・読み込みできること
      - 車両ごとにアクセル・ブレーキのキャリブレーションデータを保存しておき、選択した車両に応じて自動で読み込むことができる
    - 走行中に基準車速、実車速、アクセル開度、ブレーキ開度をリアルタイムでモニタリングできること
    - タイムスケジュールでペダルの動きを設定できる機能があること
      - この機能は運転ではなく、ペダルの動きを自動化するためのもの
      - 基準車速CSVを使用せずに、あらかじめ設定したスケジュールに従ってペダルを動かすことができる
    - タイムスケジュール操作と運転の両方ができる機能があること(シーケンス)
      - タイムスケジュール操作と運転の両方ができる機能があると、基準車速CSVを使用して運転する機能と、タイムスケジュールに従ってペダルを動かす機能の両方を提供できる
      - これにより、ユーザーは基準車速CSVを使用して運転することも、タイムスケジュールに従ってペダルを動かすこともできるようになる
    - 非常停止ボタンを押すと、すぐに全てのアクチュエータが原点位置に戻ること
    - 結果のログを残すこと
      - 記録周期: 100ms
      - アクティブ保持期間: 3ヶ月（PostgreSQL）
      - アーカイブ: 3ヶ月経過後にCSV+圧縮して外付けUSB SSD（`/mnt/archive/`）に自動退避
      - 自動削除: アーカイブ先がストレージ容量の80%超で古いものから自動削除（容量はTBD）
      - ストレージ構成: OS+PostgreSQLは内蔵NVMe SSD、アーカイブは外付けUSB SSD
    - 運転モデルの学習は"学習モード"または、過去のログから行うことができること
    - アクチュエータの動作を手動でできる機能があること
- **Should**
    - GUIでロボットの動作を細かく制御できること
    - ロボットの動作をカスタマイズできる柔軟性
    - ドキュメントやAPIが充実していること
    - シグナルタワー
    - オプションとして、走行しながら学習できる機能があること

## ハード構成
- 他に必要なものがあれば適宜追加すること
- PC linux-raspberry pi RAM16GB
- 電源 30W(raspberry pi用)
- 電源 100V→24V変換器(P-CON-CB用)
- 予備バッテリー(UPS) 24V側: P-CON-CB・アクチュエータ用
    - AC電源断時に home_return() が完了するまでの時間（目安30〜60秒）を保持できること
    - 必要容量はTBD（実機でP-CON-CB×2台＋アクチュエータ×2台の消費電流を実測して選定）
- P-CON-CB（アクチュエータコントローラ）2つ
- IAI製アクチュエータ 2つ
    - 車両のアクセルペダルとブレーキペダルに設置
- USB-RS485変換ケーブル 2つ（P-CON-CBとPCを接続、1本につき1台を独立接続）
    - /dev/ttyUSB0 ─ P-CON-CB (SLAVE_ID=1, アクセル)
    - /dev/ttyUSB1 ─ P-CON-CB (SLAVE_ID=2, ブレーキ)
- CAN読み取り装置: KvaserのUSB-CANインターフェース 1ch
  - シャシダイナモの実車速
- 非常停止スイッチ 2つ（GPIO並列接続、どちらを押しても同じ動作）
    - シャシダイナモ室内: 車両・アクチュエータ周辺の作業者用
    - 操作エリア: オペレータが操作しながら即停止できる用
- LANケーブル
    - GUIアプリ操作用：IPを合わせて接続し操作する
- シグナルタワー
- 予備バッテリー(UPS): Geekworm X1201
    - GPIO 6 でAC電源断を検知
    - I2C（アドレス 0x36）でバッテリー電圧・残量を監視
    - バッテリー持続時間: 最大6時間（18650×2本）、安全停止には十分

## ケーススタディ
- **ユースケース0 起動・初期化**
  - 電源ONでGUIが自動起動する（待機状態）
  - GUIが起動すると通信確認を実施する
    - /dev/ttyUSB0（アクセル用）、/dev/ttyUSB1（ブレーキ用）、CAN の接続を確認
    - いずれかが未接続・エラーの場合はGUIにエラーを表示して停止する
  - ユーザーがGUIの「初期化」ボタンを押すとアクチュエータの初期化を開始する
    - アラームリセット → サーボON の順に実行する
    - 前回正常終了していた場合は原点復帰（home_return）をスキップする
    - 前回異常終了・または初回起動の場合は原点復帰を実施する
  - 初期化完了後、GUIに「準備完了」を表示してユーザー操作を受け付ける

- **ユースケース1 初期設定**
  - 車両プロファイルの作成 保存
      - ユーザーが車両プロファイルを作成
      - プロファイル名をユーザーが入力して保存
      - 車両プロファイル一覧に追加される
  - キャリブレーションの実施
      - アクチュエータが自動で動作して、ペダルの接触位置を検出する。つまり、アクセルペダルとブレーキペダルの開度0%の位置を検出する
  - 学習運転
      - 学習運転用モードを走行(固定)
        - 学習運転専用の運転モードを走行する
        - この学習運転のログをもとに、運転モデルを学習する
        - モデルは車両プロファイルに紐づけ
  - 走行モードの読込み 保存
      - ユーザーが基準車速csvをアップロード
      - モード名をユーザーが入力して保存
      - 走行モード一覧に追加される
  - タイムスケジュールの作成 保存
      - ユーザーがタイムスケジュールを作成
      - スケジュール名をユーザーが入力して保存
      - スケジュールは基準車速csvとは別に管理される
      - スケジュールはタイムスケジュール一覧に追加される
  - シーケンスの作成 保存
      - ユーザーがシーケンスを作成
        - 例:
          1. タイムスケジュールNo.1: 0-10秒はアクセル50%、10-20秒はブレーキ30%
          2. 走行モードWLTP(基準車速に従って走行) 
          3. タイムスケジュールNo.2: 0-5秒はアクセル100%、5-15秒はブレーキ50%、15-20秒はアクセル30%
          4. 走行モードWLTP(基準車速に従って走行)
      - シーケンス名をユーザーが入力して保存
      - シーケンスはシーケンス一覧に追加される
  - 自動運転の実行
      - ユーザーが"走行モード","タイムスケジュール","シーケンス"のいずれかを選択
      - その中にWLTPなどの走行モードがあれば、プロファイルに紐づいた運転モデルと走行モードでフィードフォワード制御用の目標位置[mm] or 開度[%]を算出
      - 実行ボタンを押すと、選択した"走行モード","タイムスケジュール","シーケンス"に応じた動作が開始される
      - 実行中のステータスを確認できる
        - 例: 現在の基準車速、実車速、アクセル開度、ブレーキ開度などをリアルタイムでモニタリングできる
      - 終了後、ログが保存される
  - 再学習(ユーザーが必要に応じて実行)
      - 走行モードのログをもとに、運転モデルを再学習する
      - 再学習したモデルは車両プロファイルに紐づけられる
  - ログの確認
      - ユーザーが過去の走行ログを確認できる
      - 走行ログには、基準車速、実車速、アクセル開度、ブレーキ開度などの情報が含まれる
      - ログはグラフ化して表示することもできる

## 走行前チェック（共通）

学習運転・自動運転（走行モード・タイムスケジュール・シーケンス）・手動操作のいずれの開始前にも自動で実施する。
すべてをパスしない場合は走行を開始せず、GUIにエラー内容を表示してブロックする。

| チェック項目 | 確認内容 |
|---|---|
| 通信状態 | Modbus（ttyUSB0・ttyUSB1）・CAN が正常応答するか |
| サーボ状態 | 両アクチュエータがサーボON・アラームなしか |
| キャリブレーション | 現在の車両プロファイルでキャリブレーション済みか |
| 車両プロファイル | プロファイルと運転モデルがロード済みか |
| UPS残量 | 5V側UPS残量が20%以上か |
| アクチュエータ位置 | 両アクチュエータが原点位置にいるか |

## 作り方
- いきなりすべてのコードを書くのではなく、1つずつ段階を踏んで作ること
- いきなりコードを書くのではなく、まずはプロジェクトの構成や必要な機能を整理し、設計を行うこと
- 設計が固まったら、コードを書く前に必要なライブラリやツールを調査し、選定すること
- コーディングは設計に基づいて行い、必要に応じて設計を見直すこと
- コードを書いたら、必ずテストを行い、動作確認をすること

### Phase 1: 1つのアクチュエータを動かす
- `PconClient` クラスを実装（`AsyncModbusSerialClient` ベース）
  - 接続先: `/dev/ttyUSB0`（アクセル用）または `/dev/ttyUSB1`（ブレーキ用）を引数で指定
  - 各P-CON-CBは独立したシリアルポートに接続（マルチドロップではない）
  - `connect()` / `close()`
  - `enable_modbus()` — 0x0427 コイルON
  - `reset_alarm()` — 0x0407 ON→OFF
  - `servo_on()` / `servo_off()` — 0x0403
  - `home_return()` — 0x040B エッジ検出・HEND（0x9005 bit4）待機
  - `move_absolute(pos_mm, speed_mms)` — 0x9900-0x9908、CTLF=0x0000
  - `move_relative(delta_mm, speed_mms)` — CTLF=0x0008
- 単位変換: mm × 100 → レジスタ値（0.01mm単位）
- **テスト**: 実機接続後、原点復帰 → 絶対位置移動 → 相対位置移動を手動実行

### Phase 2: 1つのコントローラのエラー・電流値・位置を50msで監視
- `PconMonitor` クラスを実装
  - `read_status()` — 0x9000-0x9001（位置）・0x9005（ステータス）・0x9002（アラーム）・0x900C-0x900D（電流）を一括読取り
  - 50ms周期でポーリング
  - タイムスタンプ付きでコンソール出力
- **テスト**: 手動でアクチュエータを動かしながら監視値が更新されることを確認

### Phase 3: asyncio非同期でアクチュエータ動作と監視を並行実行
- `PconMonitor` を `asyncio.Task` として実行
- 制御コルーチンと監視タスクを `asyncio.gather()` で並行実行
- エラー発生時のタスクキャンセル処理
- **テスト**: 移動コマンド実行中に監視が50ms周期で途切れないことを確認

### Phase 4: シャシダイナモ車速をCAN経由で実車速として読み取る
- `CanReader` クラスを実装（`python-can` + Kvaser USB-CAN）
  - DBCファイルは既存のものを使用（実装時に `config/can/` に配置）
  - `cantools` でDBCファイルを読み込み、CAN IDからシグナルをデコード
  - `asyncio` 非同期受信ループ
  - 実車速[km/h]を非同期キューで提供
- **テスト**: シャシダイナモ接続時に実車速が正しくデコードされることを確認

### Phase 5: 安全
- `EmergencyStop` クラスを実装
  - Raspberry Pi GPIO入力監視（`asyncio` タスク）
  - 押下時: 全アクチュエータへ即座に `home_return()` を呼び出し
  - 状態を `asyncio.Event` で他タスクに通知
  - 電流値が閾値を超えた場合にも踏み過ぎによるペダル破壊を防ぐため、非常停止
- UPS連携
  - **5V側（Geekworm X1201）**: Raspberry Piのバックアップ電源
    - GPIO 6 を監視してAC電源断を検知（`asyncio` タスク）
    - I2C（アドレス 0x36）でバッテリー残量を定期取得してGUIに表示
  - **24V側（TBD）**: P-CON-CB・アクチュエータのバックアップ電源
    - ブレーキなしのため、AC断でペダルスプリングに負けてロッドが急激に引き戻されるリスクがある
    - home_return() 完了まで24Vを保持する必要がある（目安30〜60秒）
  - AC電源断検知時の処理順序: 走行中なら home_return()（24V UPSで動作）→ ログ保存 → PostgreSQL正常終了 → Raspberry Piシャットダウン

### Phase 7: キャリブレーション
- キャリブレーション中は車両プロファイルの最大開度設定を考慮しない
- アクセル・ブレーキともに同じ手順でゼロフルを実施する
  1. アクチュエータをゆっくり送り出しながらモデル電流（CNOW）の急増を検出して接触点を特定
  2. 接触点を開度0[%]（位置 min[mm]）とする
  3. さらにゆっくり踏み込み、電流が急増した位置[mm]を開度100[%]（位置 max[mm]）として記録
- アクセル: min_a, max_a を記録
- ブレーキ: min_b, max_b を独立して記録（アクセルのストロークを流用しない）

#### 電流急増の検出方法
- 移動開始後の最初のKサンプル（例: 10サンプル）でベースライン電流（無負荷時の平均）を算出
- 以降のサンプルで `現在電流 > ベースライン + 閾値[mA]` を満たしたら急増と判定
- 閾値はTBD（実機テストで調整、目安200〜500mA）

#### バリデーション
以下の条件をすべて満たさない場合、GUIにエラー内容を表示してキャリブレーションをやり直させる

| チェック項目 | 条件 | NGの意味 |
|---|---|---|
| 接触点(min)の検出 | タイムアウト前に検出できたか | 取り付け位置がずれている |
| フル位置(max)の検出 | タイムアウト前に検出できたか | ペダルが固着・動作不良 |
| ストロークの妥当性 | `5mm ≤ max - min ≤ 150mm`（TBD） | 検出値が物理的にありえない |
| 順序 | `max > min` | 検出ロジックの異常 |

- **テスト**: キャリブレーションを実行して、アクセル・ブレーキそれぞれの開度0%・100%の位置が正しく検出されることを確認。バリデーションNGのケースを意図的に発生させてエラー表示されることを確認

## P-CON-CB 主要Modbusレジスタ早見表
- その他、必要なコマンドは`docs/manuals/MODBUS(MJ0162-12A).pdf`を参照すること
| アドレス | 内容 | R/W |
|---------|------|-----|
| 0x0403 | サーボON（コイル） | W |
| 0x0407 | アラームリセット（コイル） | W |
| 0x040B | 原点復帰 HOME（コイル、エッジ検出） | W |
| 0x0427 | Modbus操作権有効化（コイル） | W |
| 0x9000-0x9001 | 現在位置（0.01mm単位、32bit符号付き） | R |
| 0x9002 | アラームコード | R |
| 0x9005 | デバイスステータス1（bit4=HEND、bit10=重故障） | R |
| 0x900C-0x900D | 電流値（mA、32bit） | R |
| 0x9900-0x9908 | 位置指令レジスタ群（PCMD/INP/VCMD/ACMD等） | W |

- CTLF: `0x0000`=絶対位置移動、`0x0008`=相対位置移動
- 単位: 位置値 = mm × 100（0.01mm単位）、速度値 = mm/s × 100
- スレーブID: アクセル用=1、ブレーキ用=2
